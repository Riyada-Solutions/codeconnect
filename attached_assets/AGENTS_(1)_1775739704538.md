# AGENTS.md — Lovable Project Coding Rules

## Architecture
- Always use a **service layer** for all data access and API calls
- Components must NEVER call REST APIs directly
- Place all API logic in `src/services/` files
- Services should be named by domain (e.g. `user_service.js`, `order_service.js`)

## Folder Structure
```
src/
├── pages/        # Route-level page components
├── routes/       # Route definitions and guards
├── components/   # Reusable UI components
├── services/     # All API/data access logic
└── hooks/        # Custom React hooks
```

## Naming Conventions
- Use **snake_case** for all files and folders (e.g. `user_profile.jsx`, `auth_service.js`)
- Use **PascalCase** for React component names inside files (e.g. `UserProfile`)
- Use **camelCase** for variables and function names

## Styling
- Use **Tailwind CSS only** for all styling
- Never use inline styles
- Never use CSS-in-JS, Styled Components, or plain CSS files unless already in the project

## Language
- Use **JavaScript only** — do not introduce TypeScript
- No `.ts` or `.tsx` files

## Routing
- Always use **React Router** for navigation
- Define all routes in `src/routes/`
- Use `lazy()` and `Suspense` to **lazy load all page components**

## HTTP & API
- Use **Axios** for all HTTP calls — do not use native fetch
- Create a single Axios instance with base configuration in `src/services/api.js`
- All API calls must go through `src/services/`
- Always handle API errors in the service layer and throw meaningful error messages

## Environment Variables & Secrets
- Always store API URLs and keys in `.env` files
- Never hardcode secrets, tokens, or API keys in the codebase
- Always reference the base API URL from a constant or environment variable (e.g. `VITE_API_BASE_URL`)

## Components
- Keep all components under **150 lines**
- Always wrap page-level components with an **error boundary**
- Always handle **loading and error states** visibly in the UI (e.g. spinner, error message)
- Never leave loading or error states unhandled

## Forms
- Manual validation is acceptable — no specific form library required
- Always show clear validation messages to the user

## User Feedback
- Use **toast notifications** (e.g. `react-hot-toast`) for all success and error messages
- Never use browser `alert()` or `confirm()`
- Keep toast messages short and actionable

## Comments
- Only add comments for **complex or non-obvious logic**
- Do not comment self-explanatory code
- Do not add section headers or decorative comments
